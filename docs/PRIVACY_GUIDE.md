# 公开展示与隐私边界指南（GitHub Pages/公开仓库怎么控风险）

结论先说清楚：

- **只要网站是纯前端静态站点（HTML/CSS/JS）并公开访问，那么访问者在技术上总能拿到这些前端资源**（查看源代码、抓包下载）。
- **GitHub Pages 不能实现“只能看网页、看不到源码”这种强隔离**。
- 你能做到的是：减少公开信息、避免把敏感材料放进仓库/站点、把真正敏感的能力放到“面试现场按需展示”。

---

## 1. 主动展示什么？什么留到必要再展示？什么不要展示？

### 1.1 强烈建议主动展示（加分）

- 2–3 个重点项目的：架构图 + 指标表 + demo（可选）
- 你负责的模块边界与接口（体现系统/RTL 能力）
- 验证与调试闭环（你怎么证明正确、怎么定位 bug）
- 工程化交付：README/脚本/复现步骤

### 1.2 需要时再展示（面试官问到再掏出来）

- 更细的波形截图、时序报告原图、资源报表原图
- 更完整的设计文档/测试用例

### 1.3 不要公开展示（风险高）

- 账号密码、token、内部地址、NDA 项目细节
- 简历里的敏感信息（手机号/住址/证件）
- 包含证书编号、身份证/手机号的奖状原图（建议打码后再放）

---

## 2. 公开仓库意味着什么？别人能下载我整个项目吗？

是的。

- **GitHub 仓库如果是 Public，任何人都能克隆/下载整个仓库**（包含你提交的所有文件：网页源码、图片、视频、PDF、文档等）。
- 即使你不在网页里链接某个文件，只要它在公开仓库里，一般都能被别人通过仓库下载拿到。

---

## 3. 能不能做到“别人只能看网页，看不到我的源码”？

对纯静态站点来说，**做不到真正意义上的“看不到源码”**：

- 网页要在浏览器里运行，就必须把 HTML/CSS/JS 发到用户的浏览器；用户当然可以保存/查看这些内容。
- 所以你能做的是“**不把敏感信息放进前端**”，而不是“隐藏前端代码”。

你可以把目标换成更现实的两类：

1. **让别人看不到你仓库里更多的材料**（工程文件、原始图片/PDF、笔记等）
2. **让公开网页里不包含任何敏感内容**（就算被查看源代码也安全）

---

## 4. 实用方案（按你想保护的东西选）

### 4.1 最推荐：源材料私有 + 发布内容最小化

做法：把“完整工程/文档/原始素材”放在私有仓库，只把“用于展示的网站成品”放到公开仓库/Pages。

- 私有仓库：放全部内容（包括不想公开的 docs、原图、证据截图、完整简历版本等）。
- 公开仓库：只放能公开展示的静态网页与已脱敏素材。

优点：能最大程度减少别人下载到的内容。

### 4.2 两仓库 + 自动部署（更工程化）

- `profile-private`（Private）：源代码 + 全量资料
- `profile-public`（Public）：只承载 GitHub Pages 的产物（`index.html`、压缩后的 `styles.css`、`script.js`、脱敏后的 `assets/`）

通过 GitHub Actions 从私库构建/同步到公库。

注意：即使这样，**公开网页的 HTML/CSS/JS 仍然可被查看**，只是你避免了公开“多余资料”。

### 4.3 不用 GitHub Pages：上“可控访问”的托管

如果你想“只有给链接的人才能看”，你需要**带访问控制的托管**（例如密码、登录、白名单）。

- GitHub Pages 不支持原生的访问控制。
- 这类能力通常需要：反向代理、身份验证（如 Cloudflare Access）、或私有托管平台。

### 4.4 无论如何都必须做的：站点脱敏清单

- 联系方式：主页不要放手机号/住址/证件；邮箱也可以用专门求职邮箱。
- 简历：对外只放“公开版 PDF”（可隐藏手机号/住址）；完整版只面试单发。
- 证据材料：报告/波形/奖状图片一律打码（编号、姓名、学校证件号、二维码）。
- 项目：避免放 NDA/公司项目、避免放可反推内部信息的目录/截图。
- Secrets：仓库里不要出现任何 token/密码；也不要把云盘直链里带 token 的 URL 写进代码。
